# SDK Analyzer

## Input

This skill accepts a single argument: the Azure SDK package name (e.g., `azure-ai-projects`, `azure-appconfiguration`, `azure-mgmt-appconfiguration`). The package name is passed as `{package-name}` throughout this workflow.

## Prerequisites

- The current working directory MUST be the root of the `azure-sdk-for-python` repository.
- The target SDK package must exist under the `sdk/` directory.

Verify by confirming `sdk/` exists in the current directory before proceeding. If not, stop and tell the user to navigate to the repo root.

## Step 1: Locate the SDK Package

1. Search for the package directory:
   ```
   find sdk/ -maxdepth 2 -type d -name "{package-name}"
   ```
2. Handle the result:
   - **No match** — Stop and report an error: the package was not found under `sdk/`.
   - **Multiple matches** — List all matches and ask the user to choose one.
   - **One match** — Use it and record the path as `{package-path}`.

3. Find the namespace root by locating the generated code directory:
   ```
   find {package-path} -type d -name "_generated" | head -1
   ```
   The parent directory of `_generated/` is the `{namespace-root}`.

   **If `_generated/` is not found:** The package may use inline code generation (newer pattern). Fall back to deriving the namespace root from the package name — convert hyphens to path separators (e.g., `azure-ai-projects` → `{package-path}/azure/ai/projects`). Verify this directory exists. If it does not, stop and report an error.

   When `_generated/` is absent, generated files are identified by the header comment `"Code generated by Microsoft (R) Python Code Generator"` rather than by directory location. This affects file classification in Step 2.

## Step 2: Scan and Classify Files

1. List all `.py` files under `{namespace-root}`, excluding any `tests/` directories and `_version.py`.

2. Classify each file into one of these categories:
   - **Generated** — inside `_generated/` directory, OR (if no `_generated/` exists) files containing the header `"Code generated by Microsoft (R) Python Code Generator"`
   - **Patch** — filename is `_patch.py` (outside `_generated/`)
   - **Init** — filename is `__init__.py`
   - **Utility** — everything else that is not generated, not `_patch.py`, not `__init__.py`, not `_version.py`

3. Check each `_patch.py` for emptiness using the criteria defined in `analysis-dimensions.md` (read that file from this skill's directory for the exact rules). A patch file is empty if it only contains the copyright header, the default docstring, an empty `__all__` list, and an empty `patch_sdk()` function.

4. Record counts:
   - Number of non-empty patch files
   - Number of utility files
   - Total non-blank lines across all customization files (non-empty patches + utilities)

## Step 3: Analyze by Dimension

1. Read `analysis-dimensions.md` from this skill's directory.
2. Follow its instructions to analyze all 7 dimensions:
   - Dimension 1: Client Layer
   - Dimension 2: Model Layer
   - Dimension 3: Operations Layer
   - Dimension 4: Public API Surface
   - Dimension 5: Utilities & Policies
   - Dimension 6: Async Parity
   - Dimension 7: Import Dependencies & Regeneration Risks
3. For each dimension, read the actual source files using the Read tool. Do not guess or infer file contents.
4. Record findings in the format specified by each dimension's record format. Dimension 7 findings are critical — they drive the post-regeneration verification steps and named patterns reference in the generated skill.

## Step 4: Determine Complexity and Generate

1. Read `skill-templates.md` from this skill's directory.

2. Apply the complexity threshold:
   - If customization files <= 3 AND total non-blank lines <= 300 → **single-file mode**
   - Otherwise → **multi-file mode**

3. Check if the output directory already exists:
   ```
   ls {package-path}/.github/skills/{package-name}/ 2>/dev/null
   ```
   If it exists, warn the user that existing files will be overwritten, then proceed.

4. Create the output directory:
   ```
   mkdir -p {package-path}/.github/skills/{package-name}/
   ```

5. Generate files using the templates from `skill-templates.md`, filling all placeholders with the analysis results from Step 3.

6. Key generation rules:
   - The generated skill MUST make it unmistakably clear that auto-generated files MUST NOT be edited. This warning must appear in: the critical warning block at the top, the File Map section, and the Rules section.
   - Use actual code snippets extracted from the SDK source files. Never fabricate examples.
   - Reference real file paths, class names, and method names found during analysis.
   - Generate the Post-Regeneration Verification section using Dimension 7 findings: import verification commands from the import map, ApiVersion checks if an ApiVersion enum was found, operation wrapper guidance from Dimension 3, model/enum impact checklist from monkey-patches and import dependencies.
   - Generate the Customization Patterns Reference section using the named patterns from Dimension 7. Each pattern gets a subsection with description, location, generated dependencies, code example, and verification step.
   - Omit sections/sub-files for dimensions where the analysis found nothing.
   - **Never omit** these sections regardless of findings: the critical warning, File Map (including the `_generated/` warning), Post-Regeneration Verification (at minimum Steps 1, 5, 6, 7), Async Parity, Rules.
   - In multi-file mode, do not generate sub-files for empty dimensions.

## Step 5: Report

When generation is complete, report:

1. **Mode** — Which mode was used (single-file or multi-file) and why.
2. **Generated files** — List every file created, with its full path.
3. **Summary of findings** — Brief summary of what customizations were found across each dimension.
4. **Output path** — The full path to the generated skill directory: `{package-path}/.github/skills/{package-name}/`
5. ** Skill name** — The name of the generated skill, which should be `{package-name}`.
