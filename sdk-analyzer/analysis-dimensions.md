# Analysis Dimensions for Azure Python SDK Packages

## SDK Package Structure

The SDK package has this general structure:

```
{package-root}/{namespace}/
  __init__.py
  _patch.py              # Top-level client customizations
  _*client*.py           # Hand-written client files (if they exist outside _generated/)
  _*.py                  # Other hand-written utility modules
  _generated/            # AUTO-GENERATED — do not analyze for patterns, only use as reference
    __init__.py
    _client.py
    _configuration.py
    models/
    _operations/
    aio/
  models/
    __init__.py
    _patch.py            # Model customizations
    _models.py           # May be hand-written model definitions
  operations/
    __init__.py
    _patch.py            # Operations customizations
  aio/
    __init__.py
    _patch.py            # Async client customizations
    operations/
      __init__.py
      _patch.py          # Async operations customizations
```

## How to Identify Customization Files

**File classification rules:**

- Files inside `_generated/` are auto-generated. Use them only as reference to understand the base API. Never treat them as customization patterns.
- If no `_generated/` directory exists, the package uses inline code generation. Generated files are identified by the header comment `"Code generated by Microsoft (R) Python Code Generator"`. Treat these the same as files inside `_generated/` — reference only, not customization patterns.
- Files outside `_generated/` (or non-generated files when using inline generation) that are NOT `__init__.py`, `_version.py`, or empty stubs are customization files. These are the analysis targets.
- A `_patch.py` file is **empty** (no customizations — skip it) if it ONLY contains:
  - The copyright header
  - The docstring "Customize generated code here..."
  - `__all__: List[str] = []` with no additional entries
  - An empty `patch_sdk()` function
  - No other class or function definitions
  If `__all__` contains entries or the file defines classes/functions beyond `patch_sdk()`, it is **non-empty**.

**Missing scan targets:** If a scan target file or directory does not exist in the package (e.g., no `operations/` directory), record it as absent for that dimension and move on. Do not treat a missing file as an error.

## Dimension 1: Client Layer

**Scan targets:** `_patch.py` (top-level), any `_*client*.py` outside `_generated/`, `aio/_patch.py`

**What to extract:**

1. Custom client classes — name, base class, and purpose
2. Constructor customizations — added/modified `__init__` parameters, default overrides, credential wrapping
3. Method overrides — methods that replace generated behavior, noting what changed and why
4. New methods — methods not present in generated client, with signatures and purpose
5. Pipeline policy injection — which custom policies (discovered in Dimension 5) are added to the client pipeline, and at what position
6. Credential handling — custom credential types, token scope overrides, authentication flow changes

**Record format:**

```
Client: {ClassName}
  Base: {base_class or "standalone"}
  Constructor changes: {list}
  Overridden methods: {name -> change summary}
  New methods: {name -> signature and purpose}
  Policies: {list of custom policies}
  Credential handling: {description or "default"}
```

## Dimension 2: Model Layer

**Scan targets:** `models/_patch.py`, any non-generated model files in `models/` (e.g., `models/_models.py` outside `_generated/`)

**What to extract:**

1. Custom model classes — name, inheritance chain, discriminator value if polymorphic
2. Added fields — fields not present in the generated model, with types and defaults
3. Field transformations — property wrappers, computed fields, type coercions
4. Serialization customizations — custom `_serialize` / `_deserialize` methods, attribute maps
5. `__all__` modifications — symbols added to or removed from the public model surface

**Record format:**

```
Model: {ClassName}
  Base: {base_class}
  Discriminator: {value or "none"}
  Added fields: {name: type = default}
  Transformations: {field -> description}
  Serialization: {custom behavior or "default"}
__all__ additions: [list]
__all__ removals: [list]
```

---

## Dimension 3: Operations Layer

**Scan targets:** `operations/_patch.py`, `aio/operations/_patch.py`

**What to extract:**

1. Overridden operations — method name, what changed vs generated version
2. New operations — method name, signature, purpose (not present in generated ops)
3. LRO customizations — custom polling methods, result extraction, status checks
4. Paging customizations — custom `by_page` behavior, continuation token handling, result mapping
5. `__all__` modifications — operations classes added to or removed from exports

**Record format:**

```
Operation: {method_name}
  Type: {override | new}
  Sync file: {present | absent}
  Async file: {present | absent}
  LRO custom: {description or "none"}
  Paging custom: {description or "none"}
  Change summary: {what and why}
```

## Dimension 4: Public API Surface

**Scan targets:** `__init__.py` at each level (top-level, `models/`, `operations/`, `aio/`, `aio/operations/`). Compare each with the corresponding `_generated/__init__.py` when available.

**What to extract:**

1. Re-exports from `_patch.py` — symbols imported from patch files and exposed publicly
2. Symbol renaming — cases where an internal name is re-exported under a different public name
3. Additional exports — symbols in `__init__.py` that do not come from `_generated/`
4. Removed exports — symbols present in `_generated/__init__.py` but absent from the public `__init__.py` (whether actively deleted or passively omitted — record all such symbols)

**Record format:**

```
File: {relative __init__.py path}
  Patch re-exports: [symbol list]
  Renames: {internal_name -> public_name}
  Additional exports: [symbol list with source]
  Removed exports: [symbol list]
```

## Dimension 5: Utilities & Policies

**Scan targets:** All `.py` files outside `_generated/` that are NOT `_patch.py`, `__init__.py`, or `_version.py`

**What to extract:**

1. Module inventory — list each file with a one-line purpose summary
2. Custom policies — classes inheriting from `*Policy` or `*SansIOHTTPPolicy`, with behavior description
3. Utility functions — public helper functions with signatures and purpose
4. Constants and enums — module-level constants, string enums, configuration values
5. Cross-references — which other dimensions reference or depend on these utilities

**Record format:**

```
Module: {filename}
  Purpose: {one-line summary}
  Classes: {name -> base, purpose}
  Functions: {name -> signature, purpose}
  Constants: {name -> value or description}
  Used by: [list of dimensions/files that import from this module]
```

## Dimension 6: Async Parity

**Compare:** Each sync customization file against its async counterpart.

Sync-to-async file mapping:
- `_patch.py` <-> `aio/_patch.py`
- `operations/_patch.py` <-> `aio/operations/_patch.py`
- Any `_*client*.py` <-> `aio/_*client*.py`

**What to extract:**

1. Parity status — for each pair: `identical`, `equivalent` (same logic, async keywords), `divergent`, or `missing`
2. Content comparison — if divergent, list specific differences (extra methods, different logic, missing overrides)
3. Shared code — utilities or base classes used by both sync and async layers
4. Discrepancies — any async-only or sync-only customizations that break parity

**Record format:**

```
Pair: {sync_file} <-> {async_file}
  Status: {identical | equivalent | divergent | sync-only | async-only}
  Differences: {list or "none"}
  Shared code: {list of shared modules/classes}
```

## Dimension 7: Import Dependencies & Regeneration Risks

This dimension identifies the fragile links between customization files and generated code — the exact points that break when regeneration changes the generated API. This data drives the post-regeneration verification steps in the generated skill.

**Scan targets:** All non-empty `_patch.py` files and all utility files (from Dimension 5).

**What to extract:**

1. **Import map** — For each customization file, list every symbol imported from generated modules (`_generated/` directory or files containing the generated code header). Record the exact import statements. These are the imports that can break after regeneration.
2. **ApiVersion enum** — Detect if a hand-maintained `ApiVersion` enum or similar version constant exists outside `_generated/`. Record its location, all defined version members, and the current default value (`DEFAULT_VERSION` or equivalent). If none exists, record as absent.
3. **Monkey-patching** — Detect patterns where customization code modifies generated objects at runtime: adding class attributes, patching staticmethods onto generated enums, adding backward-compatible aliases (e.g., `GeneratedEnum.OldName = GeneratedEnum.NEW_NAME`). These break silently if the generated target is renamed or removed.
4. **Shared helpers** — Functions or classes defined in one customization file and imported by other customization files. Especially note sync helpers imported by async code (these must not be duplicated). Record the defining file and all importing files.
5. **Hand-authored classes** — Classes in customization files that do NOT subclass any generated class. These are entirely custom implementations (e.g., buffered senders, custom iterators, custom error types). They still may import generated types and break if those types change.
6. **Constructor reordering** — Detect where a custom client class changes the parameter order of `__init__` relative to its generated base class (e.g., swapping `credential` and `index_name` for better ergonomics).
7. **Wire format helpers** — Functions that construct or transform wire-format request/response data (e.g., encoding parameters into pipe-delimited strings, Base64-encoded continuation tokens, custom serialization of search metadata fields). These break when the wire protocol or model schema changes.
8. **Named patterns** — Identify and name each distinct customization pattern found across all dimensions. A pattern is a reusable technique applied one or more times (e.g., "Polymorphic Delete", "Custom Pagination", "Enum Aliases", "Field Builders"). For each pattern, record: the name, a one-sentence description, which files implement it, and what generated symbols it depends on.

**Record format:**

```
File: {relative path}
  Generated imports: [{module}: {symbol1, symbol2, ...}]

ApiVersion:
  Location: {file path or "not found"}
  Members: [{version_name} = "{version_string}", ...]
  Default: {DEFAULT_VERSION value or "N/A"}

Monkey-patches:
  {target_class}.{attribute} = {value/description} (in {file})

Shared helpers:
  {function_name} defined in {file}, imported by [{file1, file2, ...}]

Hand-authored classes:
  {ClassName} in {file} — {one-line purpose}

Constructor reordering:
  {ClassName} in {file}: generated({param_order}) → custom({param_order})

Wire format helpers:
  {function_name} in {file} — {description of encoding/transformation}

Named patterns:
  {PatternName}: {one-sentence description}
    Files: [{file1, file2, ...}]
    Depends on: [{generated_symbol1, generated_symbol2, ...}]
```
